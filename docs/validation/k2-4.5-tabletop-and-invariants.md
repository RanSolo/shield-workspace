# K2-4.5 Tabletop and Invariant Validation

These tabletop cases validate the constitution and formal specification without
implementing runtime behavior. Every scenario is replayed conceptually against
the same kernel rules.

## Adapter-neutral baseline

Mission `m-23` has current subject `change-23` at immutable revision `r2`.
Technical requirement `req-fitz-r2` requires Fitz approval. Product/domain
requirement `req-simmons-r2` exists only when explicitly included by mission
policy. Human principals have valid, sequence-bounded trusted bindings.

| Scenario | Input | Required result |
| --- | --- | --- |
| Valid Fitz approval | Exact requirement, seat, subject, `r2`, verdict, binding, provenance, and sequence | Fitz requirement `satisfied` |
| Missing evidence | No Fitz record or waiver | `waiting`; never inferred from silence |
| Malformed evidence | Unknown field or missing provenance | Not satisfaction; fail closed with stable reason |
| Wrong seat | Coulson submits a Fitz technical-review record | `seat_mismatch`; `waiting` |
| Stale evidence | Valid Fitz approval targets `r1` after `r2` becomes current | `stale`; `waiting` |
| Inherited evidence | Another mission or subject has an approval | Mission/subject mismatch; `waiting` |
| Duplicate evidence ID | Two journal entries reuse one evidence ID | Replay `invalid` |
| Conflicting evidence | Active approved and changes-requested records cannot be uniquely resolved | `conflicting`; readiness `blocked` |
| New revision | `r3` explicitly supersedes `r2` | `r2` approval becomes stale for current readiness |
| Valid waiver | Authorized human submits exact revision-bound waiver | Requirement `waived`; no approval record created |
| Unauthorized waiver | Adapter, agent, model, runner, or unauthorized human submits waiver | `waiver_unauthorized`; fail closed |
| Expired waiver | Evaluated sequence exceeds explicit expiration | `pending` or `blocked` from other evidence; never waived |
| Delivery failure | Request to Fitz cannot be delivered | Communication `failed`; evidence remains `pending` |
| Execution completed | Runner says the effect completed without evidence | Execution may project completed; readiness does not become ready |
| Adapter cache disagrees | Host displays approved while replay says waiting | Kernel replay wins |

## Cross-host tabletop

Use the exact baseline requirement and decision through each transport:

| Transport | Candidate source | Kernel-relevant facts | Expected meaning |
| --- | --- | --- | --- |
| GitHub | Review or comment | Valid human binding, exact requirement/revision, verdict, provenance | Same satisfaction rules |
| Jira | Approval action | Same fields | Same satisfaction rules |
| VS Code | Signed review command | Same fields | Same satisfaction rules |
| Manual/no adapter | Trusted operator entry | Same fields | Same satisfaction rules |

Changing only the transport MUST NOT change validation, evidence status,
readiness, permission, failure reasons, or replay output. Host-native labels,
emoji, ticket states, comment delivery, or UI affordances carry no additional
authority.

## Runner and model tabletop

Repeat the baseline using different runners and models. Each receives the same
kernel permission outcome for the same journal sequence. A runner that reports an
approval on behalf of Fitz, or a model prompted to act as Coulson, produces no
human evidence. A runner that cached `allow` for `r2` must re-evaluate after `r3`
is journaled.

## Projection independence checks

| Change | Governance | Execution | Readiness | Communication |
| --- | --- | --- | --- | --- |
| Approval request delivered | unchanged | unchanged | waiting | delivered |
| Delivery fails | unchanged | unchanged | waiting | failed |
| Fitz approves exact revision | unchanged | unchanged | may become ready | unchanged |
| Runner starts | unchanged | running | unchanged | unchanged |
| Coulson pauses mission | paused | independently derived/handled | evidence statuses unchanged | unchanged |
| New revision journaled | unchanged | unchanged | prior evidence stale | unchanged |

## Replay checks

The following must produce replay failure rather than a partial projection:

- missing, duplicate, or non-contiguous journal sequences;
- duplicate event or evidence IDs;
- unsupported schema or event types;
- unknown fields in closed records;
- impossible or cyclic supersession chains;
- evidence before its requirement or trusted binding exists;
- binding validity that does not cover the evidence sequence;
- ambiguous current revision or requirement set; and
- two histories claiming the same authoritative sequence.

Replaying identical journal bytes under a different clock, locale, network,
adapter, runner, model, or host must return byte-equivalent canonical projection
data and deterministically ordered reasons.

## Invariants

- **I1 — Human exclusivity:** no non-human component can create human authority.
- **I2 — Exact binding:** satisfaction requires exact mission, requirement, seat,
  subject, revision, evidence kind, and allowed verdict.
- **I3 — Current revision:** prior-revision evidence never satisfies a new revision.
- **I4 — Explicit change:** waiver, revocation, revision, and supersession exist
  only as valid journal events.
- **I5 — Fail closed:** missing, malformed, unknown, ambiguous, inherited, stale,
  or conflicting data never becomes satisfaction.
- **I6 — Projection separation:** governance, execution, readiness, and
  communication never substitute for each other.
- **I7 — Host neutrality:** replacing a host or adapter cannot change kernel meaning.
- **I8 — Runner neutrality:** replacing a runner or model cannot change kernel meaning.
- **I9 — Replay purity:** replay performs no environmental reads and depends only
  on validated ordered journal data.
- **I10 — Journal authority:** delivery or cached host state never outranks replay.
