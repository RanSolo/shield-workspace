# SHIELD V0.3 Product Contract

- **Status:** Draft for architecture review
- **Mission:** V0.3-1 / `RanSolo/shield-workspace#24`
- **Revision:** r3
- **Authority:** Contract-definition work only
- **Implementation:** Not authorized

This contract describes the intended V0.3 release. It does not claim that the
current repository already satisfies these promises and it is not an
implementation plan.

## 1. Audience

V0.3 is for a maintainer adopting SHIELD into an existing GitHub-hosted
JavaScript or TypeScript repository.

The adopter must not need to understand SHIELD's internal architecture, private
file layout, seat prompts, or development history to install it and complete the
supported workflow.

GitHub is the first supported host. It is not part of SHIELD's identity or
kernel semantics.

## 2. Product principles

### Supported is not the same as possible

V0.3 promises only behavior explicitly included in this contract. Other
workflows may be technically possible but remain unsupported.

### Intentional public surface

Only documented commands, configuration, package exports, files, and workflows
are supported. Internal paths are not public APIs.

### Host neutrality

Every GitHub operation passes through the host-neutral adapter boundary.
GitHub behavior cannot define authority, readiness, evidence satisfaction, or
mission completion.

Seats request technical or product/domain review. Hill selects the configured
review surface and the adapter performs delivery. Fitz and Simmons do not own or
encode a transport.

### Fail closed

Missing, malformed, stale, ambiguous, inherited, conflicting, or unsupported
data cannot produce authority, readiness, or publication permission.

### Explicit human authority

Coulson, Fitz, and conditional Simmons remain human-only authority seats in
V0.3. No model, runner, adapter, host, or prompt can synthesize their decisions.

### Contracts before runtime

Public behavior is defined before implementation. Runtime choices may change
without changing adopter-visible meaning.

### Procedural guidance is not security

Prompts, seat instructions, and documentation may guide behavior but are not
security boundaries. Enforced permissions come from validated runtime policy.

## 3. Authority boundaries

V0.3 uses this rule:

> Mission approval permits planning. Wheels Up permits implementation and
> bounded draft-review publication within the approved implementation mission.
> `review.publish` permits bounded publication for review-only missions.

These permissions remain distinct even when Wheels Up carries the bounded
publication effects needed to deliver its own implementation artifacts.

### Mission approval

Mission approval authorizes the specified planning, reconnaissance,
architecture, and contract-definition work. It does not authorize repository
publication or implementation.

### Review publication

For a review-only mission that does not receive Wheels Up, `review.publish` is
explicitly authorized by Coulson for an exact set of review artifacts and
repository-relative paths. It may permit only the bounded effects needed to
create or update the configured review surface:

- reuse or create the authorized review branch;
- commit only the authorized artifacts;
- push the authorized branch;
- create or update a draft review pull request or configured review item;
- publish approved review findings; and
- request the named human review.

Review publication does not authorize implementation, runtime changes, feature
development, review approval, merge, deployment, or release.

Changes outside the authorized artifact paths fail closed. A review-only pull
request remains draft until Fury approves its exact revision for human review.

### Wheels Up

Wheels Up authorizes implementation within the approved Mission Brief. For
artifacts produced by that implementation mission, it also authorizes the
bounded branch creation, commits, push, and creation or update of a draft review
pull request needed to publish them. It does not authorize changes outside the
approved scope, marking the pull request ready before Fury approves its exact
head, merge, deployment, or release.

## 4. Supported product surface

### Installation

The supported installation unit is an exact-version npm-compatible
`@shield/team-system` package artifact installed as a development dependency.
The distribution channel and publishing administration are outside this
contract. V0.3 documentation provides one complete supported installation
command using the selected release artifact.

Unpinned installations are unsupported.

### Public package API

Consumers may rely only on documented package entry points and accompanying
JavaScript or TypeScript contracts. The minimum supported API covers:

- configuration validation;
- mission records and governance;
- mission journals and deterministic replay;
- mode references;
- human-evidence requirements and readiness;
- permission decisions; and
- the host-adapter candidate envelope.

Deep imports into undocumented package paths are unsupported.

### Configuration

The stable repository configuration location is `.shield/config.json`.

Configuration uses an explicit schema version, rejects unknown fields and
unsupported versions, identifies the repository and selected host adapter,
records supported seats and modes, references trusted human-seat bindings
without embedding credentials, and defines journal and generated-artifact
locations.

Forward compatibility is not inferred. An unsupported schema fails closed until
a documented explicit migration is performed.

### Durable repository state

SHIELD stores mission journals and generated mission artifacts under `.shield/`.
Mission journals are authoritative durable evidence. Cached host state, chat
history, or adapter delivery state cannot replace them.

The documentation distinguishes tracked configuration and review artifacts from
ignored journals, reports, temporary state, and reproducible output. Uninstall
preserves journals and reports by default.

### Supported seats

AI-operated workflow seats:

- Maria Hill — coordination;
- Daisy Johnson — reconnaissance;
- Nick Fury — architecture; and
- Melinda May — implementation.

Human authority seats:

- Phil Coulson — mission and publication authorization;
- Leo Fitz — required human technical review; and
- Jemma Simmons — product/domain review when explicitly required.

V0.3 does not support AI or hybrid occupancy of Coulson, Fitz, or Simmons.

### Supported modes

- Delivery Mode is supported end to end for the V0.3 acceptance mission.
- Debugger Mode is supported for bounded investigation and supervised
  correction.

Daily Briefing, Hotfix Mode, testing modes, and domain-mode packs are outside the
V0.3 contract.

### Supported command surface

V0.3 provides documented equivalents of:

```text
shield init
shield doctor
shield mission begin
shield mission status
shield mission step
shield mission pause
shield mission resume
shield mission cancel
shield mission report
shield evidence record
```

`mission step` advances at most one supervised cycle and is not an autonomous
loop. `evidence record` requires a trusted human binding; local execution alone
cannot establish human authority.

### First supported host

GitHub is the first supported host adapter. It may publish Mission Briefs,
review artifacts, architecture findings, and status projections; create or
update draft PR workspaces; request candidate human evidence; read candidate
review evidence; preserve source and revision references; and report delivery
success or failure.

It cannot decide authority, evidence satisfaction, readiness, execution
permission, mission completion, merge, deployment, or release. A documented
manual evidence path remains available when GitHub is unavailable.

## 5. Supported workflow

The supported V0.3 adoption workflow is:

1. Install the exact SHIELD package version.
2. Run `shield init` in an existing repository.
3. Review generated configuration and repository changes.
4. Run `shield doctor`.
5. Begin a supervised Delivery Mode mission.
6. Review and explicitly authorize the Mission Brief.
7. Create the durable mission journal.
8. For an implementation mission, advance bounded supervised implementation
   only after Wheels Up and publish its in-scope artifacts to a draft review
   surface under that same authority. For a review-only mission, create or
   update the draft review surface only after Coulson authorizes
   `review.publish` for the exact artifacts.
9. Keep the review surface draft while implementation or architecture review is
   incomplete.
10. Validate the exact pull-request revision.
11. Fury reviews that exact revision for architecture.
12. Record Fury's architecture-review decision without overwriting prior
    decisions.
13. If Fury requests changes, keep or return the pull request to draft and route
    the next action to the responsible seat.
14. When a new commit is produced, treat every prior Fury approval as historical
    and require a new architecture review for the new revision.
15. After Fury approves the current revision, Hill may use the mission's
    applicable bounded publication authority—Wheels Up or `review.publish`—to
    publish Fury's approved findings, mark the pull request ready for human
    review, and request Fitz.
16. Obtain Fitz human technical review.
17. Obtain Simmons review only when explicitly required.
18. Produce the mission report and reviewed pull request.
19. Return merge, deployment, release, and other consequential actions to the
    authorized human maintainer.

SHIELD completes its supported workflow at a review-ready pull request. It does
not merge the pull request.

## 6. Architecture-review record

Every Fury decision is an append-only architecture-review record tied to the
exact pull-request head revision reviewed.

The record captures:

- mission identifier;
- stable architecture-review identifier;
- pull-request identity and exact head revision;
- reviewer seat (`fury`);
- verdict (`approved` or `changes_requested`);
- reasons for the verdict;
- summary findings and immutable references to inline findings;
- decision timestamp and provenance;
- the seat responsible for the next action; and
- whether the pull request remains draft or must be returned to draft.

Every inline finding identifies a repository-relative file and source location
so the host adapter can place it beside the relevant diff. Host-native comment
identifiers are retained in the existing delivery/effect receipt or adapter
audit data. They never modify the append-only Fury review record, and transport
details do not change the finding's meaning.

Records are never edited to represent a later review. A new revision receives a
new append-only record under the same review requirement. Prior records remain
mission history and cannot satisfy the current revision.

Fury approval is advisory architecture evidence, not Fitz human approval and not
publication authority. Hill may advance to Fitz only when both conditions hold:

1. Fury has an `approved` record for the current exact revision; and
2. the mission's applicable bounded publication authority—Wheels Up or
   `review.publish`—remains valid for the required artifacts and surface.

## 7. Supported guarantees

An adopter can rely on these observable behaviors:

- repeated initialization does not duplicate configuration or silently alter
  unrelated files;
- diagnostics identify missing or incompatible requirements;
- unsupported configuration and contract versions fail closed;
- a recorded mission replays consistently within its documented schema;
- governance, execution, readiness, and communication remain distinct;
- human evidence binds the required seat, subject, and exact revision;
- a new revision makes prior revision-bound approval stale;
- every Fury verdict remains in append-only mission history;
- only current-revision Fury approval permits advancement to Fitz;
- review publication cannot authorize implementation;
- silence, delivery, host state, or model output cannot become human approval;
- host-adapter failure leaves evidence pending;
- one supervised step performs at most the bounded authorized work;
- restart does not repeat a completed effect;
- uncertain effects stop for human direction;
- pause, resume, cancel, status, and report preserve mission history;
- required Fitz review cannot be replaced by automated pre-review;
- uninstall preserves journals and reports by default; and
- merge, deployment, release, destructive migration, credentials, and
  production access remain outside V0.3 authority.

## 8. Explicit non-goals

The absence of a feature from this contract must not be interpreted as a future
commitment.

V0.3 does not support multiple production host adapters, background scheduling,
open-ended autonomy, general model routing, AI or hybrid human-authority seats,
seat-prompt customization as a product API, Mission Control, Oracle, Mack,
testing-mode design, domain-mode packs, plugin architecture, custom third-party
adapters, package-publishing administration, automated merge/deploy/release,
destructive migrations, policy self-modification, automatic configuration
migration, or compatibility with undocumented package paths.

V0.3 also defers generic review-publication schemas, new publication event
families, a transport registry, multiple-surface fallback, generalized
repository-mutation capabilities, retries, notifications, and publication
expiry.

## 9. Adoption acceptance criteria

A fresh external GitHub-hosted JavaScript or TypeScript repository demonstrates:

- exact-version package installation using documented instructions;
- initialization without unrelated file changes and safe repeated init;
- actionable doctor results and secret-free configuration;
- a durable supervised Delivery Mode mission;
- explicit Coulson mission authorization;
- a separately authorized review-only draft PR containing only exact authorized
  artifact paths while Wheels Up remains withheld;
- rejection of an unauthorized code or artifact change under `review.publish`;
- separate governance, execution, readiness, and communication status;
- an append-only Fury `changes_requested` record for an exact revision;
- preservation of that record after May creates a new revision;
- a new Fury review record for the new revision;
- Fury approval of the current revision before the PR becomes ready for Fitz;
- publication of Fury's file-and-location findings beside the correct diff;
- Fitz human technical review and conditional Simmons waiting;
- prior human evidence becoming stale after a revision change;
- GitHub delivery failure leaving readiness pending;
- successful trusted manual evidence fallback;
- at least one bounded supervised mission step;
- restart without repeating a completed effect;
- pause, resume, report, and cancel behavior;
- no automated merge, deployment, release, or destructive action;
- upgrade, rollback, and uninstall documentation; and
- recorded installation friction and human interventions.

Passing this scenario is necessary but does not authorize release. Coulson
retains the final V0.3 release decision.
