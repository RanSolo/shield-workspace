# K2-4.5 Human Authority, Evidence, Readiness, and Waiting Specification

This specification is normative. `MUST`, `MUST NOT`, `SHOULD`, and `MAY` have
their usual requirements meaning. It refines
[ADR 0001](../adr/0001-define-shield-kernel-boundary.md) without changing the
current runtime contracts.

## 1. Concepts and identifiers

- A **mission** is governed work identified by `missionId`.
- A **subject** is the exact artifact or decision under review, identified by
  `subjectId` and `subjectKind`.
- A **revision** is an immutable version of a subject, identified by `revisionId`.
  A mutable branch name, ticket state, or file path is not a revision identifier.
- A **seat** is a kernel-defined authority role, not a model persona or host role.
- A **trusted binding** associates one human principal with one authority seat in
  a bounded context.
- A **requirement** declares evidence needed for a specific mission subject and
  revision.
- A **record** is a candidate human decision submitted for one requirement.
- The **journal sequence** is the total order used for deterministic replay.

All identifiers are opaque, non-empty strings. The kernel MUST compare them for
exact equality and MUST NOT derive identity from display names.

## 2. Human authority seats

The closed initial seat set is:

| Seat | Authority |
| --- | --- |
| `coulson` | Authorize, edit, pause, resume, reject, or cancel a mission and approve explicit waivers within mission policy. |
| `fitz` | Issue human technical-review verdicts for requirements assigned to Fitz. |
| `simmons` | Issue human product/domain verdicts only when a mission declares a Simmons requirement. |

Coulson, Fitz, and Simmons are human authority seats. Agents, models, runners,
adapters, service accounts, and bots MUST NOT occupy, simulate, proxy, inherit,
or fabricate these seats. Simmons being conditional means no Simmons evidence is
required unless a revision-bound requirement explicitly names `simmons`.

### 2.1 Trusted binding

A trusted binding is a closed record:

```text
{
  schemaVersion, bindingId, humanPrincipalId, seatId,
  scope: { missionId | "*" }, validFromSequence, validThroughSequence | null,
  attestedBy, provenanceRef
}
```

Rules:

- `seatId` MUST be in the kernel seat set.
- `humanPrincipalId` MUST identify a human in a configured trust domain.
- `attestedBy` MUST identify the human authority or administrative trust policy
  that established the binding; self-assertion is insufficient.
- Validity is determined at the evidence record's journal sequence.
- Expired, future, revoked, ambiguous, or multiply conflicting bindings fail
  closed.
- Adapters MAY verify external identity, but only the kernel decides whether the
  submitted binding satisfies these rules.

## 3. Closed evidence contracts

### 3.1 Evidence requirement

An evidence requirement contains exactly:

```text
{
  schemaVersion, requirementId, missionId, subjectKind, subjectId, revisionId,
  requiredSeatId, evidenceKind, allowedVerdicts, createdAtSequence,
  supersedesRequirementId | null
}
```

Initial `evidenceKind` values are `mission_authorization`, `technical_review`,
and `product_domain_review`. `allowedVerdicts` is a non-empty closed subset of
`approved`, `changes_requested`, and `rejected`, further restricted by evidence
kind. Unknown fields, kinds, seats, verdicts, or schema versions are malformed.

### 3.2 Evidence record

An evidence record contains exactly:

```text
{
  schemaVersion, evidenceId, requirementId, missionId, subjectKind, subjectId,
  revisionId, seatId, evidenceKind, verdict, humanPrincipalId, bindingId,
  provenance: { adapterId, sourceRef, capturedAt }, journalSequence
}
```

The record MUST bind the requirement, required seat, evidence kind, mission,
subject, exact revision, verdict, human principal, trusted binding, provenance,
and journal sequence. Host content MAY be retained outside the kernel as audit
material, but `sourceRef` MUST be immutable enough to retrieve or correlate it.

### 3.3 Satisfaction

A record can satisfy a requirement only when:

1. both records validate as closed supported schemas;
2. `requirementId`, `missionId`, `subjectKind`, `subjectId`, `revisionId`,
   `seatId`, and `evidenceKind` exactly match;
3. the verdict is permitted by the requirement;
4. the trusted binding is valid for the human, seat, mission, and sequence;
5. the requirement is active and not superseded at that sequence; and
6. no unresolved conflicting record exists for the same requirement and
   revision.

Approval of one subject, revision, mission, requirement, or seat MUST NOT be
inherited by another.

## 4. Evidence status

For a selected current revision, every requirement deterministically derives one
of these statuses:

- `pending`: no decisive valid record or waiver exists.
- `satisfied`: a valid approving record exists with no later conflict.
- `blocked`: the latest decisive valid record is `changes_requested` or
  `rejected`.
- `stale`: a structurally relevant record targets an older revision.
- `conflicting`: multiple active, valid records cannot produce one verdict under
  the ordering rules, including duplicate sequence claims.
- `superseded`: an explicit later event replaced this requirement or evidence;
  it cannot satisfy current readiness.
- `waived`: an active valid waiver explicitly suspends the requirement.

Malformed or unknown candidates are rejected before append and never become
satisfaction. If malformed or unsupported data is present in authoritative
journal bytes, replay is `invalid` and produces no partial readiness projection.
Ambiguous, inherited, or binding-invalid candidates fail closed and cannot alter
the last valid projection. `conflicting` is reserved for incompatible valid
records that prevent one deterministic evidence verdict.

## 5. Revision and supersession

A subject revision is introduced by an explicit journal event that names its
immutable `revisionId` and, when applicable, `supersedesRevisionId`. After that
event:

- requirements for the prior revision remain historical;
- prior approvals become `stale` for current-readiness purposes;
- no adapter or runner may copy them to the new revision; and
- current readiness is derived only from requirements bound to the new revision.

Requirement or evidence replacement also requires an explicit supersession event:

```text
{
  schemaVersion, eventId, missionId, sequence, type: "evidence.superseded",
  targetKind, targetId, replacementId, reason, actorHumanPrincipalId,
  actorBindingId, timestamp
}
```

The target and replacement MUST share the mission and compatible subject scope.
Cycles, self-supersession, missing targets, future targets, or ambiguous chains
are invalid and fail closed.

## 6. Waivers

A waiver is an explicit journal event:

```text
{
  schemaVersion, eventId, missionId, sequence, type: "evidence.waived",
  waiverId, requirementId, subjectId, revisionId, authorizedSeatId,
  humanPrincipalId, bindingId, reason, scope, expiresAtSequence | null,
  timestamp
}
```

Waiver rules are deterministic:

- The mission policy MUST explicitly allow the named human seat to waive that
  evidence kind; there is no default waiver authority.
- The binding MUST be valid at the waiver sequence.
- Mission, requirement, subject, and exact revision MUST match.
- `reason` and `scope` MUST be explicit non-empty values.
- Expiration is sequence-based, not wall-clock inference.
- A waiver is revoked or replaced only by an explicit later journal event.
- A new subject revision makes an older-revision waiver stale.
- A waiver changes evidence status only; it does not manufacture an approval
  record or change governance or execution.

A revocation is also a closed journal event:

```text
{
  schemaVersion, eventId, missionId, sequence,
  type: "evidence.waiver_revoked", waiverId, requirementId,
  subjectId, revisionId, humanPrincipalId, bindingId, reason, timestamp
}
```

The revoking human MUST hold waiver authority for the same mission policy,
requirement, subject, and revision. Unknown waivers, mismatched scope, invalid
bindings, or revocation at or before the waiver sequence invalidate replay.

## 7. Independent projections

The same authoritative journal produces four separate projections:

| Projection | Meaning | Examples |
| --- | --- | --- |
| Governance | Whether the mission is proposed, approved, paused, rejected, or cancelled | Coulson decisions |
| Execution | What the runner has done | not-started, running, blocked, failed, completed |
| Readiness | Whether current revision-bound evidence permits a named action | ready, waiting, blocked, invalid |
| Communication | Whether requests or notices were delivered | queued, delivered, failed, unknown |

No projection substitutes for another. In particular, delivered communication is
not approval; approved governance is not technical readiness; completed execution
is not evidence satisfaction; and a delivery failure is not rejection.

## 8. Readiness and waiting

Readiness is a pure kernel function of the replayed journal, the named action,
and the selected current subject revision.

```text
deriveReadiness(journal, actionId, subjectId, revisionId) -> {
  state: "ready" | "waiting" | "blocked" | "invalid",
  requirementStatuses[], reasons[], evaluatedThroughSequence
}
```

Derivation order:

1. Fail `invalid` if journal validation or replay is not unique and complete.
2. Fail `invalid` if the requested subject revision is not the explicit current
   revision or the applicable requirement set cannot be uniquely selected.
3. Derive every applicable requirement status through the evaluated sequence.
4. Return `blocked` if any active requirement is `blocked` or `conflicting`.
5. Return `waiting` if any active requirement is `pending` or `stale`.
6. Return `ready` only if every active requirement is `satisfied` or `waived`.

`superseded` historical requirements are excluded only when a valid current
replacement is present. An empty requirement set is ready only when mission
policy explicitly declares that the action requires no evidence; otherwise it
fails closed as `invalid`.

Waiting is therefore derived, not commanded. A runner MAY wait for new journal
entries, but timers, polling, and notifications are outside the kernel. Silence,
elapsed time, a missed message, or an unavailable adapter MUST NOT change
readiness.

## 9. Permission decision

Before an external effect, the kernel combines—not conflates—projections:

- governance must permit the action;
- readiness for the exact current revision must be `ready`;
- the requested permission must be allowed for the runner and effect class; and
- the decision must name `evaluatedThroughSequence` and the exact revision.

The closed outcome is `allow`, `wait`, or `deny`, with stable reason codes. A
runner MUST discard an `allow` decision if the journal advances in a relevant
way or the revision changes before the effect begins.

## 10. Journal and replay semantics

The journal is the sole authoritative mission record. Candidate host data becomes
mission truth only after kernel validation and append at one sequence.

Replay MUST:

1. validate supported schema versions and closed fields;
2. require one mission, unique event IDs, and a contiguous total sequence;
3. reject chronology, subject, revision, seat, binding, and requirement mismatch;
4. apply trusted-binding changes before evaluating later evidence only;
5. apply explicit revision, waiver, revocation, and supersession events in
   sequence order;
6. derive all four projections without environmental reads; and
7. return the same value and reason ordering for the same journal bytes.

Duplicate evidence IDs, duplicate sequence numbers, non-contiguous sequences,
unsupported events, or conflicting authoritative histories make replay invalid.
Arrival order, host timestamps, API ordering, and adapter cache state have no
authority over journal order.

## 11. Stable failure reasons

An implementation MUST distinguish at least:

`malformed`, `unsupported_schema`, `unknown_field`, `unknown_seat`,
`unknown_evidence_kind`, `unknown_verdict`, `missing_requirement`,
`mission_mismatch`, `subject_mismatch`, `revision_mismatch`, `stale_revision`,
`seat_mismatch`, `binding_missing`, `binding_invalid`, `binding_ambiguous`,
`provenance_missing`, `sequence_invalid`, `duplicate_evidence`,
`conflicting_evidence`, `supersession_invalid`, `waiver_unauthorized`,
`waiver_expired`, and `requirements_undefined`.

Reason ordering MUST be deterministic and MUST NOT include host-specific meaning.

## 12. Host-neutral candidate contract

Every adapter submits the same conceptual envelope:

```text
{
  adapterContractVersion, adapterId, candidateId, candidateKind,
  missionId, subjectId, revisionId, humanPrincipalId | null,
  bindingId | null, sourceRef, capturedAt, payload
}
```

The kernel returns `accepted` with the assigned journal identity, or `rejected`
with stable reasons. GitHub is the first planned host integration, not a special
case. Jira, VS Code, Teams, Slack, Zoom, manual entry, and future adapters use the
same contract and cannot alter kernel semantics.

## 13. Implementation hold

This specification defines required meaning but does not authorize runtime
implementation or modify journal schema v1. A future implementation mission must
version the journal additions, preserve existing governance and execution
semantics, and prove the validation invariants before merge.
